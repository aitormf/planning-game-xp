# User Data Model ‚Äî Centralized User Management

## Overview

Users are managed centrally in `/users/{encodedEmail}` in Firebase RTDB. This is the **single source of truth** for user identity, roles, and project assignments.

There is no backward compatibility layer. Previous paths (`/data/developers`, `/data/stakeholders`, `/data/allowedUsers`) are deprecated and will be removed after migration.

### Key: `{encodedEmail}`

Email encoded for Firebase RTDB keys:
- `@` ‚Üí `|`
- `.` ‚Üí `!`
- `#` ‚Üí `-`

Example: `mfosela@geniova.com` ‚Üí `mfosela|geniova!com`

## ID Generation

### Developer IDs (`dev_XXX`)

- Auto-generated when a user is first assigned the **developer** role in any project.
- Format: `dev_` + 3-digit zero-padded number (e.g., `dev_001`, `dev_042`).
- Generated by reading all existing `developerId` values in `/users/`, finding the max, and incrementing.
- Once assigned, the ID never changes ‚Äî it follows the user across all projects.
- If a user is removed from all projects as developer, their `developerId` is **retained** (not deleted).

### Stakeholder IDs (`stk_XXX`)

- Same logic as developer IDs, with `stk_` prefix.
- Auto-generated when first assigned the **stakeholder** role.
- Retained permanently once assigned.

## Derived Behaviors

### `allowed` Custom Claim

- **Derived, not stored in RTDB.**
- A user gets `allowed: true` in Firebase Auth Custom Claims if they have **at least one project** with `developer: true` OR `stakeholder: true`.
- If all project assignments are removed ‚Üí `allowed` is revoked.
- Managed by Cloud Functions (not by database rules).

### Project Developer/Stakeholder Lists

- `/projects/{projectId}/developers` and `/projects/{projectId}/stakeholders` are **no longer the source of truth**.
- To get developers for a project: query `/users/` and filter by `projects/{projectId}/developer === true`.
- The EntityDirectoryService reads from `/users/` directly.

## Database Rules

```json
"users": {
  ".read": "auth != null",
  ".write": "auth != null && auth.token.isAppAdmin === true",
  "$encodedEmail": {
    ".validate": "newData.hasChildren(['name', 'email'])",
    "projects": {
      "$projectId": {
        ".validate": "newData.hasChildren(['developer', 'stakeholder'])"
      }
    }
  }
}
```

- **Read**: Any authenticated user (needed by EntityDirectoryService).
- **Write**: Only SuperAdmin (`isAppAdmin === true`).
- **Validation**: User must have `name` and `email`. Each project assignment must have `developer` and `stakeholder` booleans.

## Gmail Normalization

For Gmail/Googlemail addresses, dots in the local part are ignored by Google (the "dot trick"). When looking up users:

- `jorge.casar@gmail.com` and `jorgecasar@gmail.com` are the same person.
- The `onAuthUserCreated` Cloud Function normalizes Gmail addresses before lookup.
- The `/users/` key should use the **normalized** form for Gmail addresses.

## Migration from Legacy Structure

### Previous paths (deprecated)

| Path | Replacement |
|------|-------------|
| `/data/developers/{dev_XXX}` | `/users/{encodedEmail}` with `developerId` field |
| `/data/stakeholders/{stk_XXX}` | `/users/{encodedEmail}` with `stakeholderId` field |
| `/data/allowedUsers/{encodedEmail}` | Derived from project assignments in `/users/` |

### Migration script

`scripts/migrate-users-to-centralized.cjs` reads from legacy paths and creates `/users/` entries. Run with `--dry-run` first.
